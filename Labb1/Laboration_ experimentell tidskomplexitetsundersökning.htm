Laboration: experimentell tidskomplexitetsundersökning

Den här laborationen innehåller ingen avancerad algoritmkodning, utan är till för att man ska bli lite varm i kläderna i att experimentera med algoritmer, och fundera över en algoritms tidskomplexitet. Det är bäst att jobba två och två, men tillåtet att vara ensam eller tre. (Faran med att vara tre är att inte alla hänger med på allt, och får sämre förutsättningar för att ta till sig resten av kursen, men jag lämnar det avgörandet åt er.)

Börja gärna i förväg för att vara säkra på att ni har en chans att bli klara under laborationen. Arbeta och experimentera själva så mycket som möjligt, men fråga handledare om ni kör fast och inte vet hur ni ska komma vidare.

Algoritmen vi ska experimentera med är implementerad i följande metod: 

public static String filterOutChar(String in, char charToGetRidOf) {
    String out = "";
    for (int i = 0; i < in.length(); i++) {
        char c = in.charAt(i);
        if (c != charToGetRidOf) {
            out += c;
        }
    }
    return out;
}
  1. Läs koden och klura ut vad metoden gör.
  2. Fundera på vilken tidskomplexitet algoritmen bör ha, det vill säga hur körtiden borde öka med längden på strängen in (låt oss kalla längden NN). Ser det ut att vara lineärt? Mindre än lineärt? Mer än lineärt, kanske N2N2 eller N3N3? Varför? Det krävs inte ett slutligt svar här, men försök få en uppfattning.

  3. Testa metoden i en minimal Javaklass. Här är ett förslag: 

    public class TestFilter {
    // Här kan metoden stå.

    public static void main(String[] args) {
        String s = "Testar. Ett, två, tre";
        String t = filterOutChar(s, 'e');
        System.out.println("Efter: " + t);
    }
}

    Vad borde hända när man kör detta? Testa, gärna med 'e' utbytt mot något annat också.

  4. Mät tiden för att köra metoden. Exempelvis stoppar man in long before = System.currentTimeMillis(); före metoden, long after = System.currentTimeMillis(); efter den, och skriver sedan ut skillnaden mellan after och before. Dela med 1000.0 för att få det i sekunder.

  5. Tiden för den där teststrängen bör ha varit nästan omätbart kort. Vi vill mäta tiden för längre strängar. Vi har en fil som heter bible-en.txt som är drygt fyra miljoner tecken lång, det bör räcka mer än väl.

    Hur får vi in filen i en sträng i programmet? Den sortens kod är helt ok att leta upp på nätet och kopiera. (Till skillnad från, förstås, algoritmer som det ingår i kursen att öva sig på att implementera.) Googlar man t.ex. på ”java read file into string” hittar man kanske till exempel det här svaret på stackoverflow.com, och gör så här: 

    byte[] encoded = Files.readAllBytes(Paths.get("bible-en.txt"));
String s = new String(encoded, "US_ASCII");

    Klasserna Files och Paths ingår i paketet java.nio.file i Javas standardbibliotek, men måste importeras i koden, exempelvis genom att man skriver import java.nio.file.Files; överst i filen, ovanför klassen.

    De där anropen genererar också exceptions som man måste hantera, men eftersom vi inte är intresserade av att göra en massa felhantering för detta experiment kan vi smita undan det genom att skriva throws Exception efter main-metodens parameterlista, alltså mellan ) och {.

    Men vi vill inte bara köra programmet på en så lång sträng, vi ska ju ha olika längder! Låt oss till exempel börja med att bara låta s vara de första hundra tecknen i filen. Det kan vi fixa genom att ge två argument till i String-konstruktoranropet: new String(encoded, 0, 100, "US-ASCII").

    Börja så, och öka sedan successivt längden, till exempel genom att dubbla varje gång. Notera tiden för varje längd. För att se trenden i tidskomplexiteten behöver man fortsätta till rejält stora indata. Fortsätt öka och se om vi kan komma upp i körtider på några minuter; sedan kan vi nöja oss.

  6. Plotta grafen för tidskomplexiteten: stränglängden NN längs x-axeln och tiden längs y-axeln. Gör det antingen på papper med en linjal, eller använd till exempel ett kalkylprogram (molnbaserat eller lokalt). Ser det ut som det förväntade från steg 2?
  7. Uppskatta algoritmens tidskomplexitet utifrån det experimentella resultatet. Använd gärna dubbleringsmetoden som visades på föreläsningen, och som täcks på sidan 192–193 i boken, men det är också tillåtet att göra på andra sätt, kanske klura ut det från grafen. Försök få fram ett uttryck för att räkna ut tiden för ett visst NN, och testa om det blir rätt genom att mäta på något NN som ligger en bit från de tidigare testerna.
  8. Försök att förklara/bevisa utifrån programkoden varför den har den tidskomplexitet den verkar ha enligt experimentet.

Redovisning
